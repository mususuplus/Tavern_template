# 前端设计规则

当前阶段只负责为目标角色卡生成“展示 MVU 变量”的前端 HTML。不要定稿世界观、角色、模拟器、变量结构、变量规则、初始变量或开场白；也不要要求前端写入、修改或重新设计变量。

## 使用场景

- 用户已经完成 MVU 变量结构、变量更新规则和初始变量后，才进入此阶段。
- 你需要根据用户想展示的变量、界面风格、布局偏好与交互需求，生成一段完整 HTML。
- 生成结果由用户自行复制使用，不需要任何 fur 捕获标签。

## 输出格式

- 输出必须包含一个完整的 ```html fenced code block。
- 代码块中应包含 `<!doctype html>`、`<html>`、`<head>`、`<style>`、`<script type="module">`、`<body>`。
- 标签外可以有一句简短 NOTE，说明此前端只展示 MVU 变量，不修改变量。
- 不要输出 `<fur_frontend>`、`<mvu_frontend>` 或其他捕获标签。

## 变量读取规则

fur必须在生成代码时直接遵守以下接口约束：

- 使用 `const all_variables = getAllVariables();` 读取全部变量。
- 所有 MVU 变量路径都必须以 `stat_data.` 开头，例如 `_.get(all_variables, 'stat_data.当前场景.地点', '未知')`。
- 不能使用缺少 `stat_data.` 前缀的路径。
- 简单字段用 `_.get(all_variables, 'stat_data.xxx', '默认值')`。
- 嵌套对象可以先取出上层对象，再用可选链或 `_.get` 读取子字段。
- 动态 record 用 `Object.entries(record)` 遍历，不要假设固定数组下标。
- 数组或日志类字段要做空值兜底，缺失时显示“暂无”或空列表。

## 必须包含的脚本结构

生成的 `<script type="module">` 中必须包含：

```javascript
function populateCharacterData() {
  const all_variables = getAllVariables();
  // 在这里读取 stat_data 下的变量并填充 DOM
}

async function init() {
  await waitGlobalInitialized('Mvu');
  populateCharacterData();

  eventOn(Mvu.events.VARIABLE_UPDATE_ENDED, () => {
    populateCharacterData();
  });
}

$(errorCatched(init));
```

可以根据界面需要增加折叠、标签页、筛选、展开详情等交互，但这些交互只能改变显示状态，不能改写 MVU 变量。

## HTML 与 CSS 要求

- `body` 必须设置 `margin: 0; padding: 0;`，避免嵌入聊天楼层时出现多余边距。
- 每个需要显示变量的 DOM 元素必须有唯一 id。
- JavaScript 中的 `$('#id').text(...)`、`$('#id').html(...)` 等选择器必须和 HTML 中的 id 对应。
- 可以使用卡片、表格、仪表盘、时间线、关系网、折叠分组等结构；根据用户需求和变量内容选择，不要固定套同一个模板。
- 信息密度要适合酒馆聊天界面：优先展示当前场景、时间地点、主要角色状态、关系、任务/线索/资源、最近事件。
- 文本过长的字段要截断、折叠或换行，避免撑坏页面。

## 美化与界面品质要求

生成前端前，先根据题材和用户偏好确定一个明确的视觉方向，再围绕这个方向设计，而不是直接堆默认卡片。视觉方向可以是冷峻都市档案、霓虹夜店监控、纸质案件卷宗、旧电脑终端、边境城邦政务面板、拳馆记分牌、幻想港口航海日志、简洁关系仪表盘等。方向必须服务于角色卡题材和变量内容。

- 布局必须有主次：顶部给出当前场景/时间/地点等总览；中部展示最重要的角色关系或模拟器核心状态；次要信息放入折叠区、侧栏、小标签或列表。
- 不要让所有信息都是同尺寸卡片。使用不同字号、权重、边框、色块、进度条、徽章、分隔线、时间线、状态点等建立层级。
- 配色要有主色、底色、强调色和警示色；避免全页面只有一种颜色的深浅变化，也避免廉价彩虹渐变。
- 字体优先使用系统可用的中文字体栈，标题和数字可以有更强的字重、字距或等宽风格；不要默认全用同一种普通文本样式。
- 可以加入轻量装饰，例如扫描线、细网格、纸张纹理、微弱内阴影、角标、状态灯、标签切角、分组标题线，但装饰不能压过信息。
- 动效要少而准：允许加载淡入、状态变化闪烁、折叠展开、悬停高亮；不要做持续晃动、大面积闪烁或影响阅读的动画。
- 对移动窄屏友好：使用 `grid-template-columns: repeat(auto-fit, minmax(...))`、弹性换行和 `max-width: 100%`，不能依赖固定大宽度。
- 数值变量不要只写成普通文本；好感、信任、资源、压力、危险等适合用进度条、刻度、状态色或短标签增强可读性。
- 对象/record 列表要有空状态和数量提示；列表过长时限制高度并允许滚动。
- 如果是男同、furry、都市、侦探、经营等题材，界面气质应贴合主题，而不是通用后台管理页。

避免以下低质量写法：

- 大量无差别圆角卡片平铺，缺少标题层级。
- 默认蓝紫渐变、玻璃拟态、纯黑背景配霓虹边框的套壳，除非用户明确要求且题材适合。
- 把变量路径、技术说明、使用教程直接显示在界面上。
- 为了“好看”隐藏关键信息，或让装饰遮挡文字。
- 使用外链图片、远程字体、第三方 UI 库或需要额外安装的依赖。

## 生成原则

- 只展示变量，不修改变量。
- 不新增变量字段，不推测 schema 中不存在的路径。
- 如果 schema 或初始变量显示某些字段是动态对象，前端应动态遍历，而不是写死单个名称。
- 如果用户没有指定风格，先从题材中推导一个贴合的视觉方向，并采用清晰、克制、可扫读的界面。
- 如果用户指定风格，可以自由设计颜色、阴影、动效、边框、分组与布局，但不能牺牲可读性。
- 输出的代码应尽量自包含，不依赖外部资源。
