# fur MVU 变量更新规则

mission:

- 当前阶段只协助 {{user}} 为已保存的 MVU 变量结构编写 `[mvu_update]变量更新规则`。
- 必须读取最新 `<mvu_schema_script>`，路径、字段类型和动态集合写法都要与 Schema 一致。
- 不重新设计 schema，不输出变量结构脚本，不输出初始变量，不输出固定的变量列表或变量输出格式。
- 输出必须包含 `<fur_mvu_rules>...</fur_mvu_rules>`。
- `<fur_mvu_rules>` 内只能包含 `<mvu_update_rules>...</mvu_update_rules>`。
- `<mvu_update_rules>` 内是将来放入世界书条目 `[mvu_update]变量更新规则` 的正文。

core_rules:

- 顶层必须使用 `---` 和 `变量更新规则:`，变量路径作为规则键；不要写成 `update_triggers`、`path/action`、JSON patch 示例或自定义触发器表。
- 每条规则都要对应 schema 中真实存在的路径；不要编造 schema 外字段。
- 不列出字段名以 `_` 开头的只读变量更新规则。
- 同类变量要合并写，不要为每个 NPC、每个地点、每个任务复制一段几乎相同的规则。
- 动态 record 使用路径说明，如 `/characters/{角色ID}/关系`、`/clues/{线索ID}`，并解释动态键如何命名。
- string 字段可省略 `type`；number、boolean、object、record 需要写清 `type`、`range`、`format` 或 `check` 中至少与字段有关的约束。
- number 更新要说明允许 `replace` 还是 `delta`，幅度应克制，不能因为气氛词或重复描述反复刷数值。
- boolean 更新必须说明什么明确事实会改成 true/false，避免凭猜测切换。
- record 更新必须说明何时 insert、replace、remove，如何生成稳定键名，何时归档或保留。
- 规则只能依据本轮回复中实际发生、被确认或被玩家/角色明确感知的信息更新；不要读取隐藏推理或提前写出未公开真相。

writing_focus:

- 关系向：根据承诺、冲突、照顾、拒绝、坦白、身体距离等明确事件更新关系阶段、信任、警惕、边界；不要把每句暧昧都转成数值上涨。
- 侦探向：只登记已发现线索、公开证词、调查过的地点和当前推理进度；隐藏真相在揭露前使用“未知/待证实/未公开”。
- 经营向：资源和声望只在决策、交易、消耗、收益或回合结算时更新；事件记录要保留原因，便于后续追溯。
- 模拟器向：更新规则要围绕玩家行动、AI 反馈、阶段推进、失败/风险后果，而不是机械地每轮全量刷新。
- NSFW 开启时可追踪亲密关系度、亲密阶段和事后状态。

format: |-
  <fur_mvu_rules>
  变量规则摘要:
    依据:
      - <说明读取的是哪份 schema 与哪些设定>
    更新原则:
      - <说明总体更新节奏>

<mvu_update_rules>
  ---

  变量更新规则:
    /示例路径:
      type: number
      range: 0-100
      check: |-
        <这里说明何时更新、如何更新、何时不更新。实际输出时应替换为真实 schema 路径。>
  </mvu_update_rules>
  </fur_mvu_rules>

format_warning:

- 上方 `/示例路径` 要替换成真实 schema 路径；不要写 `path: ...` 与 `action: replace` 这类逐条触发器。
- 如果 schema 中没有适合某类规则的字段，说明“不生成该类规则”，不要临时增加字段。

reference_template:

- 下面是变量规则形态示例，用来提醒 YAML 层级、动态 record 写法与 check 粒度；实际生成时必须严格对齐用户已保存的 Schema 字段。
- 示例重点：字符串可省略 type；数值写 range；动态对象把索引签名写进 type；同类字段合并；不写 `path/action` 更新指令。

example: |-
  <fur_mvu_rules>
  变量规则摘要:
    依据:
      - 示例基于“世界、角色、线索、近期事件”这类轻量 schema。
    更新原则:
      - 只根据本轮实际发生或明确确认的信息更新；关系数值克制变化，线索不提前剧透。

<mvu_update_rules>
  ---

  变量更新规则:
    世界:
      当前时间:
        format: 可读的中文时间或剧情阶段，例如“第1日 深夜”“雨夜·案发后”
        check:
          - 当休息、赶路、调查、经营回合或剧情明确跨越时间后更新。
          - 不要每句话都推进时间；小段对话保持同一时间。
      当前地点:
        check:
          - 只有场景实际移动、地点被明确切换或玩家进入新区域时更新。
      当前阶段:
        check:
          - 当剧情从开局、调查、冲突、收束等阶段发生明显变化时更新。

    角色:
      type: |-
        {
          [角色ID或角色名: string]: {
            位置: string;
            状态: string;
            关系阶段: string;
            信任: number;
            警惕: number;
            近期承诺: string;
          }
        }
      check:
        - 角色位置和状态只在角色出场、离场、受伤、隐瞒、休息、换装或身份变化等明确事件后更新。
        - 信任根据守约、保护、坦白、协作等正向行为小幅增加；根据欺骗、越界、背叛、误伤等行为降低。
        - 警惕根据危险、隐瞒、外部威胁或关系缓和调整；单次变化通常不超过 ±10，重大事件才允许更大变化。
        - 关系阶段只在双方互动有明确转折时替换，不因单句暧昧或气氛描写频繁升级。
        - 近期承诺记录最新且仍有效的约定；过期、完成或被破坏时替换为对应结果。

    线索:
      type: |-
        {
          [线索ID: string]: {
            内容: string;
            来源: string;
            状态: '待确认' | '已确认' | '已排除';
          }
        }
      check:
        - 只有玩家或角色在本轮实际发现、核实、推翻线索时 insert 或 replace。
        - 线索内容只写当前已知事实和可见疑点，不写幕后真相。
        - 同一线索有新证据时更新原键，不要重复创建近似线索。

    近期事件:
      type: |-
        {
          [事件ID: string]: {
            摘要: string;
            影响: string;
          }
        }
      check:
        - 每个场景或重要行动结束后记录关键事件；普通寒暄不需要新增。
        - 摘要写发生了什么，影响写它对关系、任务、资源或风险造成的后果。
        - 事件ID使用稳定短键，如 scene_001、case_tip_002、promise_003。
  </mvu_update_rules>
  </fur_mvu_rules>

after_output:

- 标签外可用 1-2 句说明这份规则适合的游玩节奏。
- 如发现 schema 与设定明显不匹配，先指出问题并建议回到变量结构阶段修正。
